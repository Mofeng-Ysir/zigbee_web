# 当前仍缺的后端接口或字段

## 1. 当前入网设备需要稳定的地址来源

现状：
- `infer_result` 不带设备地址
- 前端现在只能尝试从协调器事件里推断当前正在入网的设备

建议补充其一：
- 在 `access_request / device_associated / access_decision / access_result` 事件 payload 中稳定返回 `ieee_addr`
- 或在 `infer_result` 中直接返回 `ieee_addr` 和 `short_addr`

## 2. 参考指纹图接口缺失

现状：
- 推理服务只返回当前样本的 `gaf.data`
- 弹窗右侧“对应长地址设备的指纹图”没有真实数据来源

建议新增一个独立接口，前端在点击“已知设备”或需要展示参考指纹图时，使用设备长地址 `ieeeAddr` 请求后端，后端返回该设备的双通道指纹矩阵。

### 建议接口

```text
POST /api/v1/fingerprint/reference
Content-Type: application/json
Accept: application/json
```

请求体：

```json
{
  "ieee_addr": "00:12:4b:00:de:2f:4f:3a"
}
```

### 建议返回格式

返回 `application/json`，内容为固定形状的双通道指纹矩阵。

建议返回：

```json
{
  "ieee_addr": "00:12:4b:00:de:2f:4f:3a",
  "shape": [2, 256, 256],
  "channels": [
    [
      [0.0123, 0.0188, 0.0251],
      [0.0102, 0.0179, 0.0244]
    ],
    [
      [0.1032, 0.0975, 0.0811],
      [0.1011, 0.0948, 0.0792]
    ]
  ]
}
```

字段说明：
- `ieee_addr`：请求的设备长地址
- `shape`：固定为 `[2, 256, 256]`
- `channels[0]`：第一张热力图矩阵
- `channels[1]`：第二张热力图矩阵
- 每个通道固定为 `256 x 256`
- 数值类型建议为浮点数，前端按实际 `min/max` 渲染色阶

### 前端解析结果

前端将 JSON 解析为：

```ts
type ReferenceFingerprintText = {
  ieeeAddr: string
  shape: [2, 256, 256]
  channels: number[][][]
}
```

其中：
- `channels[0]` 为第一张 `256 x 256` 热力图
- `channels[1]` 为第二张 `256 x 256` 热力图

前端收到 JSON 后，直接将 `channels[0]` 和 `channels[1]` 转成现有热力图组件所需的点阵结构即可，无需额外做文本分段解析。

### 返回失败示例

未找到该设备参考指纹时：

```text
HTTP/1.1 404 Not Found
Content-Type: application/json

{
  "code": "fingerprint_not_found",
  "message": "reference fingerprint not found"
}
```

### 这样设计的原因

- 请求只依赖 `ieeeAddr`，和当前页面上的设备主键一致
- 返回 `application/json`，字段语义明确，联调和排错更直接
- `2 x 256 x 256` 的结构固定，前端解析后可直接复用现有热力图绘制逻辑
- 后续如果要增加 `label`、`updated_at`、`source`、`min/max` 等字段，JSON 扩展成本更低
- 相比把参考图塞进其他事件接口，这个接口更独立，也更适合离线设备页和实时详情弹窗共用

## 3. 历史入网设备详情缺少当次识别的热力图快照

现状：
- 历史列表可展示记录
- 但点击历史记录后，拿不到那一次识别对应的热力图

建议补充其一：
- 在 `access_decision / access_result` 事件里增加 GAF 快照或引用
- 或新增按 `event_id` 查询识别详情的接口

建议接口：

```text
GET /api/v1/events/{event_id}/fingerprint
```

## 4. 顶部按钮缺少可调用接口

当前页面上有两个按钮，但现有文档没有对应接口：

- `重连设备`
- `开关入网`

建议补充：

```text
POST /api/v1/coordinator/reconnect
POST /api/v1/permit-join/open
POST /api/v1/permit-join/close
```

或统一为：

```text
POST /api/v1/permit-join
```

请求体示例：

```json
{
  "open": true,
  "duration_seconds": 180
}
```

## 5. 如果设备列表想稳定显示设备类型，建议直接返回 role

现状：
- 前端当前根据 `capability` 做简单推断
- 这不是严格可靠的类型来源

建议在 `/api/v1/devices` 返回：

```json
{
  "role": "Router"
}
```
